# グループ横断合意スコア（A-2）設計書

日付: 2026-07-10
状態: ユーザー承認済み（会話上で設計合意）
出典: `docs/ユーザーフィードバック実装検討.md` A-2（要望「異なるグループ間で高い賛成割合を持つ意見こそ『みんなの合意』として表示すべき」賛成42/反対4）

## 背景・目的

詳細ページ分析タブの「みんなが賛成した意見」は現在**全体賛成率順**
（`sortByTotalAgreeRate`）で並んでおり、特定グループだけが強く賛成した意見や、
一部グループしか投票していない意見が上位に来る（=多数決）。

実データ検証（2026-07-09）: 「外国人受け入れ」テーマの現行トップ3は全体100%表示
だが、実態は特定グループがほぼ投票していない意見だった。グループ横断スコアでは
「国籍問わず税・社会保険を負担すべき（3グループとも約9割賛成）」等が最上位になる。

## スコア定義（Polis本家準拠）

**group-aware-consensus**: 各グループの平滑化賛成率の**積**。

```
score(意見) = ∏(全グループg) (agrees_g + 1) / (totalVotes_g + 2)
```

- `totalVotes_g = agrees_g + disagrees_g + passes_g`（当該意見へのグループgの投票数）
- 根拠: Polis本家 polismath の `group-aware-consensus` と同一式
  （`math/src/polismath/math/conversation.clj`: `(/ (+ A 1.0) (+ S 2.0))` を `(reduce *)`）
- ラプラス平滑化 `(A+1)/(S+2)` により、未投票グループ（0票）は0.5（中立）となり
  生の0%のように不当に沈まない。少票グループの100%/0%の過大評価も抑制される
- 集約に「最小値」ではなく「積」を使う理由: 本家準拠で根拠をコードで示せる。
  （最小値=maximin案も検討したが、本家に前例がないため不採用。2026-07-10ユーザー判断）

**スコアは並び順の計算にのみ使う内部値**。画面には表示しない。

## UI方針（決定事項）

**並びだけ差し替え・UI不変**（2026-07-09ユーザー選択）。

- セクション見出し「みんなが賛成した意見」は現状のまま
- 各意見カードの表示内容（全体の賛成/反対/どちらでもない、グループ別円グラフ、
  ツールチップ）は一切変更しない
- 新しいUI要素は追加しない（「画面の情報量増加」否決(21/30)の民意に沿う）

## 実装

変更対象: `Client/public_html_app/javascript/detail.js` **のみ**
（データ・サーバー・他画面は不変）

1. `getGroupConsensusScore(item)` を新設
   - `item.groupData`（既存。extractGroupsの出力）から
     `∏ (agrees+1)/(totalVotes+2)` を計算して返す
   - `groupData` が欠損・空の場合は `null` を返す（呼び出し側でフォールバック）
2. `sortByConsensusScore(list)` を新設
   - スコア降順。スコアが `null` の要素は `totalAgreeRate` を代替キーに末尾側へ
   - 同点タイブレークは `totalVotes` 多い順（現行 `sortByTotalAgreeRate` と同じ）
3. `initializeArticles()` の「みんなが賛成した意見」生成箇所:
   - `groups.length >= 2` のとき `sortByConsensusScore(displayData)`
   - `groups.length <= 1` のとき従来の `sortByTotalAgreeRate(displayData)`
     （1グループでは横断合意が定義できないため）
4. グループ別セクション（`sortByGroupAgreeRate`）は変更しない

## エラー処理

- スコア計算は try 不要の純関数とし、数値化は既存 `toInt` を使う（NaN混入防止）
- `groupData` 欠損行があっても例外を出さず、その行だけ従来キーで並ぶ
  （分析タブ全体が死なないこと）
