# テーマ日時記録と更新順ソート 設計書

| 項目 | 内容 |
| --- | --- |
| 作成日 | 2026-07-08 |
| 対象 | Server(バッチ)+ Client(ホーム画面) |
| 目的 | テーマの作成・更新の日時を記録し、一覧を「動きのある順」で並べられるようにする |
| 制約 | 本番への影響がある操作は行わず、ローカル検証で実装を完了させる（マージ・デプロイはユーザー判断） |

## 決定事項

1. **themes.csv に3列追加（末尾）**: `created_at` / `commented_at` / `updated_at`
2. **両方の更新時刻を保持**: コメント増加の時刻（`commented_at`）と、投票または意見が増えた時刻（`updated_at`）を別々に記録し、両方ソートに使える
3. **デフォルトソートは「意見順」（`commented_at` 降順）**（投票順だと同じテーマばかり上に来るため）
4. **過去データは3列とも空白**とし、空白は「表示なし」・ソートでは末尾（既存 `create_date` 列は変更しない）
5. **カード左下に日時を表示**: `作成 <created_at>　更新 <updated_at>`。空白の項目は出さない

## データ仕様（themes.csv）

| 列 | 内容 | 記録タイミング |
| --- | --- | --- |
| `created_at` | 作成日時 | batch-create がテーマを公開した時 |
| `commented_at` | 新しい意見が投稿された日時 | batch-update でコメント数が**増えた**時 |
| `updated_at` | 投票または意見が増えた日時 | batch-update で投票数またはコメント数が**変化**した時（コメント増なら同時に更新） |

- 形式: **JSTの `YYYY-MM-DD HH:MM`**（表示にそのまま使え、文字列比較でソート可能）
- 新規作成時は3列とも作成時刻で初期化（新テーマがどのソートでも先頭に来る）
- 互換性: 旧クライアント+新CSV=余分な列は無視される／新クライアント+旧CSV=未定義→空白扱い。デプロイ順序に依存しない

## サーバー仕様（`Server/web/api/services/batch.py`）

- `THEME_HEADERS` 末尾に3列追加
- `update_themes` の変化検知を「投票数の変化 **または** コメント数の変化」に拡張（現行は投票のみ）
  - コメント数が増加: `commented_at = now` かつ `updated_at = now`
  - それ以外の変化（投票数の変化・コメント数の減少）: `updated_at = now` のみ
- 旧形式CSV（3列なし）を読んだ場合は空文字として扱い、次回書き込みで新ヘッダーに移行
- テーマ作成（batch-create）: 新規行の3列すべてに現在時刻を設定
- 削除・編集経路では3列の値をそのまま保持
- 既存ユニットテスト（`Server/web/tests/`）に、変化検知と時刻記録のケースを追加

## クライアント仕様（`Client/public_html_app/`）

- ソート選択肢: **意見順（デフォルト）**・更新順・新着順・古い順・人気順
  - 意見順 = `commented_at` 降順 / 更新順 = `updated_at` 降順
  - 空白は末尾へ。空白同士および同時刻は `id` 降順
- カードに `data-commented` / `data-updated` 属性を付与しDOMソートに使用（既存方式踏襲）
- カード左下に小さくグレーで表示: `作成 2026-07-08 12:00　更新 2026-07-09 08:30`
  - `created_at` 空白→「作成」を出さない、`updated_at` 空白→「更新」を出さない、両方空白→行ごと非表示
- 既存の投票数・コメント数表示（右下）はそのまま

## 検証（ローカル完結・本番無影響）

- サーバー: pytest（Dockerコンテナ内）で変化検知・時刻記録・旧CSV互換のユニットテスト
- クライアント: ローカル環境（localhost:8081）で
  1. 本番CSV（旧形式=3列なし）→ 日時非表示・ソートが破綻しないこと
  2. テスト用の新形式CSV（puppeteerのリクエスト差し替えで注入。nginx等のインフラは変更しない）→ 全ソート・日時表示・空白混在の並び
  3. スクリーンショットの目視確認
- 本番反映（mainマージ）はユーザー承認後。マージだけでサーバー(Lambda)・クライアントの両方が自動デプロイされ、次回バッチ書き込み時からCSVが新形式になる

- 詳細画面への日時表示
