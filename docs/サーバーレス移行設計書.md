## 1. 移行の原則

1. **現環境無変更**: カットオーバーまで、稼働中のEC2・cron・既存CloudFront/S3には一切手を入れない（Phase 0のクイックウィンのみ例外、§6.1）
2. **切り替え点はDNSのみ**: `api.pol-is.jp` の Route53 レコード（現在 Aレコード → EIP、TTL 300秒）を API Gateway へのエイリアスに変更するだけ。**切り戻しも5分で完了**
3. **二重実行の防止**: 新環境のバッチスケジューラは、カットオーバー時まで**無効状態でデプロイ**する（旧cronと新スケジューラが同時に動くと pol.is へのテーマ二重投稿が起きるため）
4. **データは移行前後で検証**: 件数照合 + サンプル突合を移行手順に組み込む

## 2. ターゲットアーキテクチャ

```
ユーザー
  │
  ├─ pol-is.jp / app.pol-is.jp ……（現状のまま）S3 + CloudFront
  │     └─ csv/*: バッチLambdaが Cache-Control: max-age=300 付きで直接更新（無効化API廃止）
  │
  ├─ contact.pol-is.jp / share.pol-is.jp ……（現状のまま）API GW + Lambda + SES
  │
  ├─ admin.pol-is.jp【新設】…… S3 + CloudFront + OAC
  │     └─ CloudFront Function でIP許可リスト検査（現 nginx の /dashboard/ IP制限を代替）
  │
  └─ api.pol-is.jp【切り替え】…… API Gateway (HTTP API) + ACM(ap-northeast-3)
        │    ※証明書はACM自動更新 = certbot運用の恒久廃止
        │
        ├─ Lambda「api」(コンテナ, FastAPI + Mangum)
        │     /theme/*  … AI テーマ生成（LangChain + OpenAI, LangSmith）
        │     /admin/*  … 下書き管理（IP許可リスト検査はアプリ内実装を継続利用）
        │     /batch/healthcheck
        │
        ├─ Lambda「batch-update」(zip, httpx)
        │     └─ EventBridge Scheduler rate(5 minutes)
        │        pol.is レポートCSV取得 → S3更新（Cache-Control付与、無効化なし）
        │
        └─ Lambda「batch-create」(コンテナ, Selenium + headless Chromium)
              └─ EventBridge Scheduler rate(15 minutes)
                 APPROVED下書きを pol.is に作成 → themes.csv 追記
                 ※15分制限超過が判明した場合は ECS Fargate スケジュールタスクに変更（§8）

データ層
  ├─ DynamoDB「drafts」(オンデマンド課金) … t_draft の移行先
  ├─ S3 app.pol-is.jp …（現状のまま）テーマ/レポートCSV。移行不要
  ├─ S3 アーカイブバケット【新設】… MySQL全ダンプ等の保全先
  └─ SSM Parameter Store (SecureString) … 全シークレットの一元管理【新設】

監視
  ├─ Route53ヘルスチェック（https://api.pol-is.jp/batch/healthcheck, 標準間隔30秒, $0.50/月）
  ├─ CloudWatchアラーム → SNS「Polis-Server-Alert」(既存流用) → メール通知
  └─ Synthetics Canary は廃止（$31/月 → $0.50/月）
```

### 2.1 コンポーネント設計詳細

#### Lambda「api」— FastAPI のLambda化

- **方式**: 既存FastAPIアプリに [Mangum](https://mangum.io/) ハンドラを追加し、コンテナイメージとしてデプロイ（LangChain系の依存が重くzip 250MB制限を超えるため）
- **変更点**:
  - `main.py` に `handler = Mangum(app)` を追加（既存のuvicorn起動と共存可能 = ローカルDocker開発は現状のまま）
  - `configs/` の環境別ファイル読み込み → **環境変数 + SSM Parameter Store** に置き換え（シークレットのコード同梱を廃止。棚卸しで確認したコミット済みシークレットの解消を兼ねる）
  - `cruds/t_draft.py` → DynamoDB版DAOに差し替え（§3.2）
  - memcached依存を削除（現状ほぼ未使用）
  - TimeoutMiddleware(300s) → Lambdaタイムアウト600sに設定し、API GWの29秒制限に注意（下記）
- **API Gateway の29秒タイムアウト問題**: AI生成（`/theme/generate_*`）は現行で最長数十秒かかる可能性がある。対策として HTTP API のタイムアウト上限緩和（2024年以降、リージョナルは29秒超に引き上げ申請可）または**クライアント側は既にステップ分割済み**（axis → comments → descriptions を個別リクエスト）のため、1リクエスト29秒以内に収まる見込み。**Phase 3 で実測し、超える場合は Lambda 関数URL（タイムアウトなし）を該当エンドポイントのみに使う**
- **コールドスタート**: LangChain読み込みで初回数秒。個人プロジェクトの利用頻度では許容とし、問題があればメモリ増量（1769MB=1vCPU）で緩和

#### Lambda「batch-update」— 投票情報自動更新

- 純Python（httpx + boto3）でzipデプロイ。Chromium不要
- 現行 `BatchService.update` と同ロジック: themes.csv取得 → 各レポートCSV取得 → 差分あればS3更新
- **変更点**: `clear_cache`（CloudFront無効化）を廃止し、`upload_bytes` に `CacheControl: max-age=300` を付与
- タイムアウト: 900秒（テーマ数131件 × 直列fetchでも十分）

#### Lambda「batch-create」— Polisへのテーマ作成（Selenium）

- コンテナイメージ（headless Chromium + chromedriver + Selenium。現行 `Server/web/Dockerfile` の構成を流用）
- メモリ2048MB、タイムアウト900秒
- **1回の起動で1下書きだけ処理する**設計に変更（現行は全APPROVEDを一括処理）。15分制限内に確実に収め、失敗時の影響を1件に限定する。残りは次回スケジュール（15分間隔）で処理
- pol.isログイン情報はSSM Parameter Storeから取得（現行のハードコードを解消）

#### DynamoDB「drafts」

- 課金: オンデマンド（現行の書き込み頻度なら実質$0）
- テーブル設計:

| 属性 | 型 | 備考 |
| --- | --- | --- |
| `id` (PK) | S | 現行 `t_draft.id` を文字列化。新規作成分はULID |
| `post_status` | N | 0=SUGGESTED / 1=GENERATED / 2=APPROVED / 3=POSTED / 101=REJECTED（現行enum踏襲） |
| `title`, `theme_name`, `theme_description`, `theme_comments`, `theme_category`, `conversation_id`, `report_id`, `origin_url` | S/N | 現行カラム踏襲。`origin_html`(LONGTEXT) は移行対象外（スクレイピング休眠機能用のため。アーカイブには残る） |
| `status`, `create_date`, `update_date` | N/S | 論理削除フラグと日時（ISO 8601文字列） |

- GSI: `post_status-index`（PK: `post_status`）— batch-create の「APPROVED一覧取得」と admin の一覧表示に使用
- PITR（ポイントインタイムリカバリ）有効化（無料枠規模では課金ほぼゼロ）

#### admin.pol-is.jp（管理画面の移設）

- 現行: `api.pol-is.jp/dashboard/`（EC2 nginx がIP制限付きで配信）
- 新: S3（非公開 + OAC）+ CloudFront + ACM。**CloudFront Function** でviewer IPを許可リストと照合し、不一致は403
  - WAFは使わない（WebACL $5/月+ でコスト目標に反する。CloudFront Functionは実質無料）
- `Server/admin/javascript/admin.js` のAPI参照先は `api.pol-is.jp/admin/*` のままで無変更。API側のIP許可リスト検査（アプリ内実装）も二重の防御として継続

#### シークレット管理（SSM Parameter Store, SecureString）

| パラメータ | 現在の置き場所（問題） |
| --- | --- |
| `/polisjapan/openai-api-key` | `configs/*/credentials.py`（コミット済み） |
| `/polisjapan/langsmith-api-key` | 同上 |
| `/polisjapan/polis-login` (user/pass) | `services/batch.py` にハードコード |
| `/polisjapan/user-access-key` `/polisjapan/batch-access-key` | `configs/*/constants.py` |
| `/polisjapan/admin-allow-ips` | 同上 |

- Lambdaは初回起動時に一括取得してキャッシュ。**移行完了時に全キーをローテーション**する（git履歴に残っている前提で扱う）

#### 監視

- Route53ヘルスチェック → `https://api.pol-is.jp/batch/healthcheck`（HTTPSで証明書検証も兼ねる）
- 失敗時: CloudWatchアラーム → 既存SNS `Polis-Server-Alert`
- **加えてバッチ失敗の検知を新設**（現行にない）: batch-update / batch-create Lambda のErrorsメトリクスアラーム → 同SNS
- カスタムメトリクス約37個は移行時に棚卸しし、原則廃止（$8/月削減）

## 3. データ移行計画

### 3.1 移行対象の整理

| データ | 現在地 | 移行先 | 方式 |
| --- | --- | --- | --- |
| `t_draft`（テーマ下書き） | EC2上のMySQL | DynamoDB「drafts」 | §3.2 |
| `t_user` / `t_account` / `t_user_add`（休眠） | 同上 | **移行しない** | アーカイブのみ |
| テーマ/レポートCSV | S3 `app.pol-is.jp` | **移行不要**（そのまま新Lambdaが読み書き） | ― |
| MySQL全体 | EC2上のMySQL | S3アーカイブバケット | §3.3 |
| EC2ディスク全体 | EBS | AMIスナップショット | 撤収時に取得（§6 Phase 5） |

### 3.2 t_draft → DynamoDB 移行手順

1. **エクスポート**（現環境・読み取りのみ）:
   `docker exec PolisJAPAN_db mysqldump ... t_draft` + `SELECT ... INTO OUTFILE` 相当のJSONエクスポートスクリプト（SSH経由で実行）
2. **変換**: 移行スクリプト（Python/boto3）で型変換（id文字列化、日時ISO化、`origin_html`除外）
3. **インポート**: `batch_write_item` でDynamoDBへ投入
4. **検証**:
   - 件数照合: MySQL `COUNT(*)` = DynamoDB `Scan` count（status別にも照合）
   - サンプル突合: 全post_status から数件ずつ全属性を比較
   - APPROVED件数が0件であることの確認（カットオーバー時の二重投稿リスク源のため。残っていれば先に旧cronに処理させ切るか、新環境で処理するかを明示的に選ぶ）
5. **差分再同期**: カットオーバー直前に `update_date > 初回移行時刻` の行だけ再抽出して上書き投入（§5 手順内）。**必ず `--delta` フラグを付けて実行する**（部分ファイルのため全件数照合をスキップし、投入行の突合のみ検証する。フラグなしだと正常でもVERIFY FAILEDになる）

### 3.3 アーカイブ（保全）

- S3バケット `polisjapan-archive`【新設】: バージョニング + SSE-S3 + 非公開 + Glacier Instant Retrievalへの移行ライフサイクル（90日）
- 格納物:
  1. `mysqldump --all-databases --single-transaction` の全量ダンプ（移行開始時と カットオーバー直後の2回）
  2. EC2上の `Server/` ディレクトリ一式（certs等の実体確認用。シークレット含むため必ず非公開バケット）
  3. 現行構成のエビデンス（棚卸しレポート、SG設定等のJSON）

## 4. Terraform 構成

```
Infra/
└── terraform/
    ├── backend.tf          # stateはS3バケット（polisjapan-tfstate, バージョニング+ロック）
    ├── environments/
    │   └── production/     # 当面はproductionのみ。tfvarsで環境差分を管理
    ├── modules/
    │   ├── api/            # API GW + Lambda「api」+ ACM + カスタムドメイン
    │   ├── batch/          # Lambda「batch-update」「batch-create」+ EventBridge Scheduler
    │   ├── data/           # DynamoDB + アーカイブS3 + SSMパラメータ(値は手動投入)
    │   ├── admin-site/     # admin.pol-is.jp: S3 + CloudFront + CF Function
    │   └── monitoring/     # Route53ヘルスチェック + アラーム + SNS購読
    └── imports/            # 既存リソースのimportブロック置き場（Phase 5で段階的に取り込み）
```

- **シークレット値はTerraformに書かない**: SSMパラメータは`aws_ssm_parameter`のガワだけ作り、値は `aws ssm put-parameter` で手動投入（tfstateへの平文残留を防ぐ）
- **既存リソースは当面Terraform管理外**: 稼働中のCloudFront/S3/Route53ゾーンには触れない。カットオーバーで変更するのは `api.pol-is.jp` レコード1本のみで、これは手動変更 → Phase 5 で `import` してコード管理化
- Lambdaコンテナ用に ECR リポジトリ ×2（api / batch-create）
- 実行にはTerraform用のIAMユーザー/ロール（AdministratorAccess相当を推奨、棚卸し用とは別）が必要

## 5. カットオーバー手順（Runbook）

前提: Phase 3 の内部テスト完了、新環境が API GW デフォルトドメインで全機能検証済み

| # | 手順 | 所要 | 切り戻し可否 |
| --- | --- | --- | --- |
| 1 | MySQL全量ダンプ #2 を取得しS3アーカイブへ | 5分 | ― |
| 2 | **旧cronコンテナを停止**（`docker stop PolisJAPAN_cron`）+ **旧nginxで `/batch/update` `/batch/create_all` を403化**（location追加してreload）— 凍結と同時に、並行稼働期間中に旧エンドポイントが叩かれて古いthemes.csvで本番を上書きする事故を遮断する | 5分 | cron再開 + nginx設定を戻すだけ |
| 3 | t_draft 差分再同期（§3.2-5、**`--delta` フラグ必須**）→ 突合確認 + **APPROVED残数=0の確認**（残っていれば「旧環境で処理し切る/新環境に委ねる」を明示的に選んでから先へ。二重投稿防止の要） | 10分 | ― |
| 4 | Route53 `api.pol-is.jp` を Aレコード(EIP) → API GWカスタムドメインへのエイリアスに変更 | 5分（TTL300） | **レコードを戻すだけ（5分）** |
| 5 | **CloudFrontの `/csv/*` を1回だけ手動無効化** — 既存キャッシュはCache-Controlなしで最大24時間残留するため、TTL方式への切替時に一度だけ掃除する（数円） | 2分 | ― |
| 6 | **batch-updateスケジューラのみ有効化**（読み取り中心・低リスク）→ 投票数がCSVに反映されることを確認 | 5分 | 無効化するだけ |
| 7 | 動作確認: healthcheck / AIテーマ生成E2E / 管理画面ログイン・承認操作（この時点ではまだ公開されない） | 30分 | ― |
| 8 | **batch-createスケジューラを有効化** → 手順7で承認したテーマが15分以内にpol.is反映→themes.csv追記されることを確認 | 20分 | 無効化するだけ |
| 9 | **MySQL残留行チェック**: `SELECT id, theme_name, update_date FROM t_draft WHERE update_date > '<手順3の時刻>'` — DNS伝播中に旧APIへ届いた投稿がないか確認。あれば該当行を `--delta` で追加移行 | 5分 | ― |
| 10 | Route53ヘルスチェックのアラームがOKであること確認 | 5分 | ― |

**切り戻し条件と手順**: 手順7〜8で致命的問題が出た場合 → Scheduler無効化(6,8) → DNS戻し(4) → nginx403解除+旧cron再開(2)。加えて**必ずDynamoDBの逆確認を行う**: DNS切替(手順4)以降に新環境へ投稿された下書きはDynamoDBにのみ存在するため、`aws dynamodb scan --table-name polisjapan-drafts --filter-expression "id >= :t" --expression-attribute-values '{":t":{"N":"<手順4のepoch秒>"}}'` で抽出し、あればMySQLへ手動INSERTしてから運用再開する（怠るとユーザー投稿がサイレント消失する）。また凍結中に新環境がpol.isに投稿済みのテーマがあれば、旧`themes.csv`とdraftステータスの整合を手動確認する。

**旧環境は切り替え後も2週間そのまま稼働**させ（cron停止・batchエンドポイント403のまま）、問題なければPhase 5で撤収。

## 8. リスクと対策

| リスク | 影響 | 対策 |
| --- | --- | --- |
| Selenium処理が15分/2GBに収まらない | batch-create不成立 | **解消済み（2026-07-04 運用者確認: 1テーマ作成は15分未満）** → Lambdaコンテナ方式で確定。万一将来超過した場合のフォールバックはECS Fargateスケジュールタスク |
| AI生成がAPI GW 29秒制限を超える | テーマ作成UX劣化 | Phase 3で実測。超過エンドポイントのみLambda関数URL直結 or 非同期化（ジョブID方式）にフォールバック |
| pol.is側のUI変更でSelenium破損 | テーマ作成停止 | 移行とは独立の既存リスク。batch-createのエラーアラーム新設で検知可能になる（現行は検知手段なし） |
| DynamoDB化でadmin画面の一覧挙動が変わる | 運用画面の不具合 | GSIで現行同等のクエリを担保。Phase 3のE2Eにadmin操作一式を含める |

| Lambdaコールドスタート | 初回リクエスト数秒遅延 | 許容。問題なら Provisioned Concurrency は使わず（$$）、メモリ増量とイメージスリム化で対応 |
| themes.csvの書き込み競合（batch-update×batch-create） | 公開直後のテーマ行消失 | **解消済み（2026-07-05）**: ETag If-Matchによる楽観ロック+競合時再取得リトライを実装（`BatchService.write_themes_csv`） |
| 下書きID上限バリデーション（le=256） | 移行後の新規下書きが削除不能 | **解消済み（2026-07-05）**: 上限をBIGINT_MAXに変更、epoch秒IDのテスト追加 |
| カットオーバー直後の既存CloudFrontキャッシュ残留 | 最大24時間古いCSVが配信 | **Runbook手順5に一回限りの手動無効化を追加**。テーマ削除はTTLで反映できないため、deleteフローに対象パス限定の無効化を実装済み |
| git履歴に残るシークレット | 漏洩リスク | Phase 5 で全ローテーション + `git filter-repo` による履歴除去（リポジトリ公開状態の確認を先に行う） |
