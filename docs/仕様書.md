## 1. サービス概要

### 1.1 背景

Polis は「ブロードリスニング」（インターネット経由で意見を募り、テクノロジーを活用して合意形成を探る取り組み）を実現するSNSシステムである。

- 「賛成」「反対」の投票を通じ、レスバトルや誹謗中傷の応酬を生まない議論ができる
- 統計的分析による意見グループの可視化で「落とし所」となる意見を探ることができる

一方、オリジナルの Polis は日本のユーザーにとってユーザービリティが低く、そのままでは日本での普及が難しい。

### 1.2 目的

**「Polis」のシステムを日本に社会実装する。**

そのために、下記の課題を解決し、Polis をコアとして SNS に求められる周辺機能を実装する。

1. 日本語による UI・操作説明がなく、日本人ユーザーが初見で受け入れ難い
2. PC 前提の UI となっており、モバイルユーザーが快適に使用できない
3. 一覧・検索機能がなくテーマを見つけることができない
4. 投稿に手間が掛かり、思いついたテーマを議題にするまでのハードルが高い

### 1.3 実装方針

投票・意見グループ化などのコア機能は本家 Polis（pol.is）をそのまま利用し、本システムは以下を提供する。

- LP によるサービス周知
- 日本語 UI・チュートリアル
- テーマ一覧・検索機能
- モバイル最適化された体験（PWA 対応）
- AI アシストによる簡易なテーマ作成
- Polis の投票データを取得・集計して独自表示する分析画面

## 3. システム構成

### 3.1 ドメイン構成

| ドメイン | 役割 | ホスティング |
| --- | --- | --- |
| `pol-is.jp` | LP | S3（バケット `pol-is.jp`）+ CloudFront |
| `app.pol-is.jp` | WEB アプリ（PWA） | S3（バケット `app.pol-is.jp`）+ CloudFront |
| `api.pol-is.jp` | API（テーマ作成・バッチ・管理） | API Gateway (HTTP API) + Lambda（コンテナ、FastAPI + Mangum）。TLS は ACM（自動更新） |
| `admin.pol-is.jp` | 管理画面 | S3 + CloudFront + CloudFront Function（IP 許可リスト検査） |
| `contact.pol-is.jp` | お問い合わせ受付 | API Gateway + Lambda + SES |
| `share.pol-is.jp` | 共有リンクリダイレクト | ―（共有 URL 生成に使用） |
| `pol.is` | 投票・意見グループ化のコア機能（本家 Polis） | 外部サービス |

AWS リージョン: `ap-northeast-3`（大阪）

### 3.2 アーキテクチャ概要

**フルサーバーレス構成**（2026-07 移行完了）。常時稼働サーバーを持たず、コストメリットと可用性を両立する。

```
ユーザー
  │
  ├─ Route 53 → CloudFront → S3 ……… LP / WEBアプリ（静的サイト・CSVキャッシュ）
  │     └─ csv/*: バッチLambdaが Cache-Control: max-age=300 付きで直接更新
  │
  ├─ API Gateway → Lambda → SES …… お問い合わせ通知送信（contact / share）
  │
  ├─ pol.is（埋め込みAPI）………………… 投票・意見投稿・意見グループ化
  │
  ├─ admin.pol-is.jp ………………………… S3 + CloudFront + CloudFront Function（IP制限）
  │
  └─ api.pol-is.jp …… API Gateway (HTTP API) + ACM
        │
        ├─ Lambda「api」(コンテナ, FastAPI + Mangum)
        │     /theme/*（AIテーマ生成）・/admin/*（下書き管理）・/batch/healthcheck
        │
        ├─ Lambda「batch-update」← EventBridge Scheduler rate(5 minutes)
        │     pol.is レポートCSV取得 → S3更新（Cache-Control付与）
        │
        └─ Lambda「batch-create」(コンテナ, Selenium + headless Chromium)
              ← EventBridge Scheduler rate(15 minutes)
              APPROVED下書きを pol.is に作成（1起動1件）→ themes.csv 追記

データ層
  ├─ DynamoDB「polisjapan-drafts」……… テーマ下書き（旧 MySQL t_draft の移行先）
  ├─ S3 app.pol-is.jp ……………………… テーマ/レポートCSV
  ├─ S3 polisjapan-archive ……………… 旧環境のMySQL全ダンプ等のアーカイブ
  └─ SSM Parameter Store (SecureString) … 全シークレットの一元管理

監視
  ├─ Route53ヘルスチェック（/batch/healthcheck）→ CloudWatchアラーム → SNS → メール
  └─ batch-update / batch-create の Errors メトリクスアラーム
```

- テーマ・投票データの実体は **S3 上の CSV ファイル**（クライアントが直接読む読み取り専用「データベース」）
- テーマ下書きは **DynamoDB**（`polisjapan-drafts`、オンデマンド課金・PITR有効）。旧ユーザーアカウント機能（休眠）はデータごと S3 アーカイブへ退避し移行対象外
- インフラ全体を **Terraform**（`Infra/terraform/`）でコード管理（既存リソースも import 済み・計115リソース）

## 4. クライアント仕様（`Client/`）

3 つの静的サイトで構成され、開発時はそれぞれ独立した nginx コンテナで配信される（`Client/docker-compose.yml`）。

| サイト | ディレクトリ | 開発ポート | 本番 |
| --- | --- | --- | --- |
| LP | `Client/public_html/` | 8080 | s3://pol-is.jp/ |
| WEB アプリ | `Client/public_html_app/` | 8081 | s3://app.pol-is.jp/ |
| 管理画面 | `Client/public_html_admin/` | 8082 | admin.pol-is.jp（S3 + CloudFront、IP 制限） |

### 4.1 LP（`public_html/`）

- `index.html` — タイトル「Polis JAPAN | 分断をなくすSNS」。セクション: `#hero` / `#problems` / `#features` / `#demo`（デモテーマカード → アプリへ誘導）/ `#info`。JSON-LD 構造化データを含む
- サブページ: `contact/`（お問い合わせ）、`terms/`（利用規約）、`policy/`（プライバシーポリシー）、`license/`（ライセンス）
- `javascript/contact.js` — お問い合わせフォーム。`https://contact.pol-is.jp/contact` へ `mail` / `name` / `content` を POST
- Google Search Console 検証ファイルを配置

### 4.2 WEB アプリ（`public_html_app/`）

PWA として登録可能（`manifest.json`、`javascript/service-worker.js`、スプラッシュ画像一式は `Asset/PWA/generate.zsh` で生成）。

#### 画面構成

| 画面 | ファイル | 概要 |
| --- | --- | --- |
| ホーム（テーマ一覧） | `index.html` + `javascript/home.js` | テーマカードの一覧表示・検索・絞り込み・ソート |
| テーマ詳細・分析 | `detail/index.html` + `javascript/detail.js` | Polis 埋め込みによる投票と、独自レンダリングの分析レポートのタブ切替 |
| テーマ作成モーダル | `javascript/create.js` | AI アシスト／手動の 2 モードによるステップ式作成ウィザード |

#### ホーム画面（`home.js`）

- `/csv/themes.csv` を取得しテーマカードを描画
- カテゴリ絞り込み（1: 社会・政治、2: お金・資産、3: 男女・性別、4: 外国人問題、5: テクノロジー、6: 医療・福祉、7: 生活、8: その他）
- キーワード検索（トークン分割・`data-searchtext` によるローカル検索）
- ソート: 更新順（デフォルト、`updated_at` 降順）・意見順（`commented_at` 降順）・新着順（ID降順）・古い順（ID昇順）・人気順（投票数）。ソートボタンは選択中のソート名を表示
- テーマカード右上に更新日時（`updated_at`）を表示（旧データ等で空白の場合は非表示）
- 画面右下にハンバーガーメニュー（お知らせ・お問い合わせ・利用規約・プライバシーポリシー）。お知らせはスライドモーダル+アコーディオン表示で、新着があれば自動表示（Cookie既読管理、`javascript/notice.js`）

#### テーマ詳細画面（`detail.js`）

- URL クエリ `?conversation_id=` を `.polis` 要素の `data-conversation_id` に設定し、`https://pol.is/embed.js` を動的ロードして投票 UI を埋め込む
- 分析レポートは iframe ではなく独自実装:
  `/csv/report/report_<conversation_id>.csv` を取得し、コメントごとの賛成・反対・パスの円グラフと意見グループ別ツールチップを描画（賛成率順ソート、円グラフタップで意見内容・投票数を表示）
- 共有ボタン: `https://share.pol-is.jp/link?conversation_id=...` を生成しクリップボードへコピー
- 「詳細レポート」ボタン: 本家 `https://pol.is/report/<report_id>` へリンク

#### テーマ作成（`create.js`）

- **AIアシストモード**: テーマ入力 → 対立軸生成（`/theme/generate_axis`）→ シードコメント生成（`/theme/generate_comments`）→ 説明文生成（`/theme/generate_descriptions`）→ カテゴリ選択 → 投稿（`/theme/post_draft`）
- **手動モード**: テーマ・対立軸・コメント（8 件以上）・説明・カテゴリを手入力して投稿
- 下書きは localStorage に自動保存
- 投稿された下書きはサーバーの `t_draft` テーブルに登録され、バッチにより Polis へ反映される（§6 参照）

#### 共通ライブラリ（`common.js`）

- `API_URL = "https://api.pol-is.jp/"`、`USER_ACCESS_KEY`（bot 対策用アクセスキー）
- CSV パーサ（`loadCsvAsJson` / `parseCSV`）、Cookie・localStorage ヘルパー、`TutorialManager`（チュートリアル）、`ModalManager`、`CustomSelectManager`、クリップボード、ローディング表示

### 4.3 管理画面（`public_html_admin/`、配信コピーは `Server/admin/`）

- 本番は `admin.pol-is.jp`（S3 非公開バケット + OAC + CloudFront）で配信。CloudFront Function が viewer IP を許可リストと照合し、不一致は 403（旧 EC2 nginx の `/dashboard/` IP 制限を代替）
- 運営者がアクセスキーを入力（Cookie 保存）し、`https://api.pol-is.jp/admin/info` からテーマ下書き一覧を取得
- 下書きの承認（`/admin/approve`）・編集（`/admin/edit`）を実施

### 4.4 スタイル（`Client/scss/`）

- 単一エントリ `style.scss` → 各サイトの `style.css` に sass コンテナ（node:20-alpine、`sass --watch`）でコンパイル
- 構成: `common/`（reset / color / font / components / layout / animation）、`pages/top/`（LP）、`pages/home/`、`pages/detail/`、`pages/admin/`
- パーシャルは `_` プレフィックスの命名規則

## 5. サーバー仕様（`Server/web/api/`）

### 5.1 技術スタック

| 項目 | 内容 |
| --- | --- |
| フレームワーク | FastAPI（async）。ローカルは uvicorn、本番 Lambda は **Mangum** ハンドラ（`main.py` の `handler = Mangum(app)`） |
| 言語 / パッケージ管理 | Python 3.11（ローカル開発）/ 3.12（Lambda コンテナ）/ Poetry |
| データ層 | **DynamoDB**（本番）/ MySQL 8.0 + SQLAlchemy 2.0（ローカル開発）。`repositories/draft.py` の DraftStore 抽象で `DATA_BACKEND`（mysql / dynamodb）を切り替え |
| AI 連携 | LangChain + langchain-openai（OpenAI GPT-5 系）、LangSmith（プロンプト管理・トレーシング）、LangGraph |
| スクレイピング | Selenium（Chromium）+ BeautifulSoup、httpx |
| Web 検索 | DuckDuckGo（ddgs） |
| AWS 連携 | aioboto3 / boto3（S3・DynamoDB） |

`main.py` では production / serverless 環境で `/docs` `/redoc` `/openapi.json` を無効化し、`TimeoutMiddleware`（300 秒）と CORS を設定。マウントされるルーターは `admin` / `batch` / `theme` の 3 つ（`user` ルーターは実装済みだが未マウントの休眠機能）。memcached 依存は移行時に削除済み。

リクエストは `core/common_route.py`（`CommonRoute`）で共通処理される: パスからルーター/API 名を解決 → 認証チェック → 非同期 DB セッション開始 → サービスの動的生成（`api.services.<router>`）→ S3 クライアント初期化 → ハンドラ実行 → 後処理。エラーは `trace_id` 付き JSON に正規化される。

### 5.2 API エンドポイント一覧

#### `/theme`（テーマ作成 API・`USER_ACCESS_KEY` で保護）

| エンドポイント | メソッド | 概要 |
| --- | --- | --- |
| `/theme/generate_axis` | POST | テーマから対立軸（立場）を AI 生成 |
| `/theme/generate_comments` | POST | 対立軸ごとのシードコメントを AI 生成 |
| `/theme/generate_descriptions` | POST | テーマ説明文を AI 生成 |
| `/theme/post_draft` | POST | ユーザー投稿の下書きを `t_draft` に登録（status: APPROVED） |

#### `/batch`（バッチ API・`BATCH_ACCESS_KEY` で保護）

| エンドポイント | メソッド | 概要 |
| --- | --- | --- |
| `/batch/update` | POST | 全テーマの最新レポート CSV を pol.is から取得し、投票数・コメント数に変化があれば S3 の `themes.csv` / レポート CSV を `Cache-Control: max-age=300` 付きで更新（CloudFront 無効化は廃止し TTL で反映） |
| `/batch/create_all` | POST | status = APPROVED の下書きを Selenium で pol.is 上に作成し、`themes.csv` に追加、status を POSTED に更新 |
| `/batch/delete` | POST | テーマを `themes.csv` から削除、レポート CSV を S3 から削除、下書きを論理削除（削除は TTL で反映できないため対象パス限定の CloudFront 無効化を実施） |
| `/batch/healthcheck` | GET | 死活監視（Route53 ヘルスチェックの監視先） |

本番の定期実行は HTTP 経由ではなく、EventBridge Scheduler が専用 Lambda（`lambda_handlers/batch_update.py` / `batch_create.py`）を直接起動する（§7 参照）。HTTP の `/batch/*` はローカル開発・手動実行用として残る。

#### `/admin`（管理 API・`BATCH_ACCESS_KEY` + IP 許可リストで保護）

| エンドポイント | メソッド | 概要 |
| --- | --- | --- |
| `/admin/info` | POST | 下書き一覧取得 |
| `/admin/approve` | POST | 下書きを APPROVED に更新 |
| `/admin/edit` | POST | 下書き内容の編集 |

#### `/user`（未マウント・休眠機能）

`mail_check` / `create` / `login`（httponly の `session_id` Cookie を発行）/ `reload` / `edit` / `delete`。パスワードは `utils/security.py` でハッシュ化。

エラーはカスタムステータスコードを使用（例: 481 InvalidAccessKey、482 PolisReportUnavailable）。

### 5.3 AI テーマ生成（`services/theme.py`）

- モデル: `gpt-5`（テーマ・対立軸生成）、`gpt-5-nano`（reasoning_effort=low、コメント・説明・カテゴリ生成）
- プロンプトは **LangSmith から実行時に取得**（キー: `get_theme` / `get_axis` / `get_axis_standalone` / `get_comments` / `get_comments_standalone` / `get_description` / `get_description_standalone` / `get_category` / `get_background_selector`）
- 生成前に DuckDuckGo News で背景情報を収集しコンテキストとして付与
- LangChain LCEL（`RunnableParallel` / `RunnableBranch` / `.abatch`）でチェーン構成

### 5.4 テーマ自動作成（`services/batch.py`）

- Twitter / Togetter のスクレイピング（Selenium + BeautifulSoup）によるテーマ候補の自動抽出パイプラインを持つ
- `create_theme_on_polis()` は Selenium で pol.is の Web UI を操作し、サインイン → カンバセーション作成 → シードコメント投入 → 可視化有効化 → レポート作成を行い、URL から `conversation_id` / `report_id` を抽出する

### 5.5 環境設定（`api/configs/`）

環境変数 `APP_ENV`（`serverless` / `production` / `development` / `localhost`、デフォルト development）で環境フォルダを選択し、共通 `constants.py` にマージして読み込む。

| ファイル | 保持する設定 |
| --- | --- |
| `constants.py` | `API_BASE_URL` / `CLIENT_BASE_URL` / `ENCRYPT_SALT` / `BATCH_ACCESS_KEY` / `USER_ACCESS_KEY` / `ADMIN_ALLOW_IPS`（CIDR）/ `DATA_BACKEND` / `LOG_ENABLE_FLAGS` / `CORS_PARAMETERS` |
| `credentials.py` | `OPENAI_API_KEY` / LangChain（LangSmith）関連 / AWS アクセスキー等 |
| `database.py` | DB ホスト・DB 名（`app_db`）・ユーザー・ポート・パスワード（ローカル MySQL 用） |

- **本番（`serverless`）**: 設定ファイルに値を持たず、すべて **Lambda 環境変数** から読む。シークレットの実体は SSM Parameter Store (SecureString) にあり、Terraform が Lambda 環境変数へ注入する（`configs/serverless/`）。`DATA_BACKEND = "dynamodb"`
- **ローカル（`localhost` / `development`）**: 2026-07-07 以降、設定ファイルはシークレットを **環境変数から読む**（ファイルに実値を置かない = コミット可能）。実値は `Server/web/.env`（gitignore 対象、雛形は `.env.example`）に置き、docker-compose が `env_file` として web コンテナに読み込む。未設定でもローカル専用のデフォルト値で起動可能

## 6. データ仕様

### 6.1 CSV（S3 バケット `app.pol-is.jp` 配下、クライアントの実質的な読み取り専用 DB）

#### `csv/themes.csv` — テーマ一覧マスタ

| カラム | 説明 |
| --- | --- |
| `id` | テーマ ID |
| `category` | カテゴリ ID（1〜8、§4.2 参照） |
| `title` | テーマ名 |
| `description` | 説明文 |
| `conversation_id` | Polis のカンバセーション ID（埋め込み・レポート CSV のキー） |
| `report_id` | Polis のレポート ID（詳細レポートへのリンク用） |
| `votes` | 総投票数（バッチで自動更新） |
| `comments` | コメント数（バッチで自動更新） |
| `create_date` | 作成日 |

#### `csv/report/report_<conversation_id>.csv` — テーマ別集計レポート

Polis の `comment-groups.csv` エクスポート形式。

```
comment-id, comment, total-votes, total-agrees, total-disagrees, total-passes,
group-a-votes, group-a-agrees, group-a-disagrees, group-a-passes,
group-b-..., …
```

### 6.2 テーマ下書き — DynamoDB「polisjapan-drafts」

本番のテーマ下書きは DynamoDB で管理する（旧 MySQL `t_draft` から 2026-07 に移行）。オンデマンド課金・PITR 有効。

| 属性 | 型 | 備考 |
| --- | --- | --- |
| `id` (PK) | S | 旧 `t_draft.id` を文字列化。新規作成分は epoch 秒ベース ID |
| `post_status` | N | 0=SUGGESTED / 1=GENERATED / 2=APPROVED / 3=POSTED / 101=REJECTED（`models/types.py::PostStatus`） |
| `title`, `theme_name`, `theme_description`, `theme_comments`, `theme_category`, `conversation_id`, `report_id`, `origin_url` | S/N | 旧カラム踏襲 |
| `status`, `create_date`, `update_date` | N/S | 論理削除フラグと日時（ISO 8601 文字列） |

- GSI: `post_status-index` — batch-create の「APPROVED 一覧取得」と admin の一覧表示に使用
- アクセスは `repositories/draft_store_dynamo.py`（DraftStore 抽象の DynamoDB 実装）経由

ローカル開発では従来の MySQL（`app_db`、`models/tables.py`）を使用（`DATA_BACKEND = "mysql"`）。休眠機能のテーブル（`t_user` / `t_account` / `t_user_add`）は移行せず、全データを S3 `polisjapan-archive` にアーカイブ済み。

### 6.3 投票情報自動更新のデータフロー

1. EventBridge Scheduler が 5 分間隔で Lambda「batch-update」を起動
2. S3 から `csv/themes.csv` をダウンロード
3. 各テーマの `https://pol.is/api/v3/reportExport/<report_id>/comment-groups.csv` を httpx で取得
4. 投票数・コメント数に変化があれば、レポート CSV と再生成した `themes.csv` を `Cache-Control: max-age=300` 付きで S3 にアップロード（`themes.csv` は ETag If-Match の楽観ロックで batch-create との書き込み競合を防止）
5. クライアントは CloudFront 経由で最新 CSV を直接取得（TTL 5 分で自動反映。無効化 API は使わない）

## 7. バッチ（EventBridge Scheduler + Lambda）

本番の定期実行は EventBridge Scheduler が専用 Lambda を直接起動する（`Infra/terraform/modules/batch/`）。

| ジョブ | スケジュール | Lambda | 内容 |
| --- | --- | --- | --- |
| batch-update | rate(5 minutes) | `lambda_handlers/batch_update.py`（api と同一コンテナイメージ・ハンドラ指定のみ変更） | 投票情報の取得・CSV 更新（§6.3） |
| batch-create | rate(15 minutes) | `lambda_handlers/batch_create.py`（Selenium + headless Chromium 入りコンテナ、メモリ 2048MB・タイムアウト 900 秒） | 承認済み下書きの Polis への公開。**1 回の起動で 1 件だけ処理**し、15 分制限内に確実に収め失敗の影響を 1 件に限定（残りは次回実行で処理） |

- 両 Lambda の Errors メトリクスにアラームを設定し、SNS 経由でメール通知（旧環境には無かったバッチ失敗検知）
- バッチ処理量は管理テーマ数に比例し、ユーザーアクセスに依存して急増減しないため、Lambda の従量課金と相性が良い
- 旧 cron コンテナ（`Server/cron/`）は旧 EC2 専用だったため削除済み（2026-07-07）。ローカルでバッチを動かす場合は `/batch/update` 等を手動で curl する

## 8. インフラ・デプロイ

### 8.1 本番構成（フルサーバーレス・Terraform 管理）

- **静的サイト**: S3 + CloudFront（LP: CF `E1L9GH70CTWAY6` / アプリ: CF `E1VDGUET2Z2OBX` / 管理: admin.pol-is.jp）、Route 53 で DNS 管理
- **API**: API Gateway (HTTP API) + Lambda ×3（api / batch-update / batch-create、§3.2）。コンテナイメージは ECR（`polisjapan-api` / `polisjapan-batch-create`）
- **TLS**: ACM（自動更新）。旧 Let's Encrypt / certbot 運用は廃止
- **シークレット**: SSM Parameter Store (SecureString) で一元管理し、Terraform が Lambda 環境変数へ注入
- **AWS 認証**: Lambda は IAM ロール、ローカルは環境変数のアクセスキー
- **IaC**: `Infra/terraform/`（modules: data / api / batch / admin-site / monitoring）。tfstate は `s3://polisjapan-tfstate`、実行は `terraform` プロファイル（terraform-deploy IAM ユーザー）。既存リソース（S3 / CloudFront / Route53 / ACM / contact・share Lambda / SES）も import 済みで計 115 リソースをコード管理。手順詳細は `Infra/terraform/README.md`
- **監視**: Route53 ヘルスチェック（`/batch/healthcheck`）+ CloudWatch アラーム（API 5xx・バッチ Errors）→ SNS → メール通知

### 8.2 開発環境

すべて Docker 上で動作（ルート `README.md` 参照）。**ローカル開発は移行前と同一構成**（FastAPI + MySQL + cron を Docker Compose で起動）。

```sh
# サーバーサイド（Swagger: http://localhost:80/docs）
cd Server && docker-compose build && docker-compose up

# クライアントサイド（LP:8080 / アプリ:8081 / 管理:8082）
cd Client && docker-compose build && docker-compose up
```

- `Server/web/Dockerfile`: ローカル開発用。python:3.11-slim + Chromium / chromedriver + Noto CJK フォント（Selenium 用）。ビルド時に pip-licenses で `THIRD_PARTY_LICENSES.html` を生成
- `Server/web/Dockerfile.lambda`: 本番 Lambda「api」「batch-update」共用イメージ（python 3.12 / AL2023 ベース）
- `Server/web/Dockerfile.lambda-chromium`: 本番 Lambda「batch-create」用イメージ（headless Chromium + chromedriver 入り）
- テーブル設計・API 設計は Google SpreadSheet + Google Apps Script で管理し、ソースへ書き出す運用（概要書より）

### 8.3 デプロイ手順

クライアントとサーバーは **GitHub Actions による自動デプロイ**（2026-07-07 導入）。インフラのみ Terraform を手動実行する。

認証はいずれも **OIDC フェデレーション**（`aws-actions/configure-aws-credentials`）で、長期アクセスキーを GitHub Secrets に置かない。Assume できるのは本リポジトリ `main` ブランチのワークフローのみ（IAM ロールの信頼ポリシーで制限。フォークからは Assume 不可）。IAM 定義は `Infra/terraform/environments/production/github_actions.tf`。

#### クライアント（静的サイト） — `.github/workflows/deploy-client.yml`

`main` への push（`Client/public_html*/**` 変更時）で自動実行: `aws s3 sync`（**`csv/*` は除外**。CSV はバッチ Lambda が管理）→ CloudFront 無効化（LP / アプリ / 管理画面）。IAM ポリシー側でも `app.pol-is.jp/csv/*` への書き込みを明示 Deny（二重の防波堤）。手動で行う場合のコマンドは `Client/README.md`。

#### サーバー（API・バッチ Lambda） — `.github/workflows/deploy-server.yml`

`main` への push（`Server/web/**` 変更時）で自動実行:

1. コンテナイメージをビルドして ECR へ push（タグ = コミット SHA）
   - `Dockerfile.lambda` → `polisjapan-api`（Lambda「api」「batch-update」が共用）
   - `Dockerfile.lambda-chromium` → `polisjapan-batch-create`
   - Lambda はマルチプラットフォームマニフェスト非対応のため `provenance: false` / `sbom: false` でビルド
2. `aws lambda update-function-code` で 3 関数を新イメージに切り替え
3. `/batch/healthcheck` で疎通確認（コールドスタート込みでリトライ）

Terraform 側は Lambda の `image_uri` を `ignore_changes` にしており、`terraform apply` が CI のデプロイを巻き戻すことはない（初回作成時のみ tfvars の値を使用）。

#### インフラ — Terraform（手動）

`Infra/terraform/environments/production/` で `terraform plan` → `apply`（プロファイル `terraform`）。シークレット値は Terraform に書かず、`aws ssm put-parameter` で手動投入する。手順詳細は `Infra/terraform/README.md`。

## 9. セキュリティ・可用性

### 9.1 設計方針（概要書より）

- ユーザーアクセスがある部分（LP・アプリ・CSV 配信）はマネージドサービス（S3 / CloudFront）で可用性を確保。アクセス増加時も柔軟にスケール可能
- API・バッチ・データ層もフルマネージド（Lambda / DynamoDB / EventBridge）。常時稼働サーバーが無く、SSH や OS パッチ等の運用対象そのものが存在しない
- データキャッシュである CSV は S3 の自動バージョニングで保護。DynamoDB は PITR（ポイントインタイムリカバリ）有効
- Web サーバーへの改竄・ハッキングリスクは静的ホスティングにより低減
- AI によるテーマ下書き生成のみ負荷があるが、アクセス比率として極小の見込み（負荷増時は別途対応）

### 9.2 実装上の保護

| 対象 | 保護方式 |
| --- | --- |
| `/theme/*` | `USER_ACCESS_KEY`（クライアント埋め込みの bot 対策キー） |
| `/batch/*` | `BATCH_ACCESS_KEY` |
| `/admin/*` | `BATCH_ACCESS_KEY` + IP 許可リスト（CIDR、`utils/security.py`） |
| 管理画面 `admin.pol-is.jp` | CloudFront Function による IP 許可リスト検査（API 側の IP 検査と二重防御） |
| 本番の Swagger | 無効化（`/docs` 等を閉鎖） |
| シークレット | SSM Parameter Store (SecureString)。Terraform には値を書かない（tfstate バケットは非公開・暗号化・バージョニング必須） |

### 9.3 注意事項（要改善）

- コード中のシークレット実値は 2026-07-07 に全廃した（configs の環境変数化、`services/batch.py` のフォールバック実値削除）。ただし **pol.is ログインパスワードは過去コミットに実値が残っており、リポジトリが公開のためローテーションが必須**（移行 Phase 5 のシークレット全ローテーションと合わせて実施。`docs/サーバーレス移行設計書.md` §6）
- `USER_ACCESS_KEY` はクライアント JS に埋め込まれるため秘匿性はない（bot 対策目的の運用であることに留意）
