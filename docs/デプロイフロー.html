PolisJAPAN デプロイフロー

サーバーレス移行後（2026-07）の本番反映手順まとめ — 詳細は docs/仕様書.md §8 / Infra/terraform/README.md

クライアント/サーバー: GitHub Actions 自動デプロイ インフラ: Terraform 手動 認証: OIDC（長期キー不使用） リージョン: ap-northeast-3（大阪）

全体像 — デプロイは3系統

A クライアント（静的サイト）

mainブランチへのpushで自動デプロイ（deploy-client.yml）。

main へ push→ GitHub Actions→ s3 sync→ CloudFront 無効化

B サーバー（API・バッチ Lambda）

mainブランチへのpushで自動デプロイ（deploy-server.yml）。

main へ push→ GitHub Actions→ buildx → ECR→ Lambda 切替→ healthcheck

C インフラ（AWS リソース）

AWS リソースの追加・変更はすべて Terraform 経由。

.tf 編集→ terraform plan→ terraform apply

A. クライアント（静的サイト）のデプロイ

対象: Client/public_html/（LP）・Client/public_html_app/（WEBアプリ）・Client/public_html_admin/（管理画面 → admin.pol-is.jp）

mainブランチにpushするだけ。対象パスの変更を検知して .github/workflows/deploy-client.yml が実行される:

  1. OIDCでIAMロール polisjapan-gha-deploy-client をAssume（S3同期とCloudFront無効化の権限のみ）
  2. 3サイトを aws s3 sync --delete（csv/* は除外）
  3. CloudFrontを3ディストリビューションとも無効化（HTML/CSS/JSの即時反映）
⚠ csv/* は絶対に同期しない S3 上の CSV はバッチ Lambda（batch-update）が5分間隔で管理する「本番データ」。ワークフローの --exclude "csv/*" に加え、IAMポリシーでも app.pol-is.jp/csv/* への書き込みを明示Denyしている（二重の防波堤）。

手動デプロイが必要な場合（Actions障害時など）は Client/README.md のコマンドを使用（--dryrun で差分確認してから実行）。

B. サーバー（API・バッチ Lambda）のデプロイ

mainブランチにpushするだけ。Server/web/** の変更を検知して .github/workflows/deploy-server.yml が実行される:

  1. OIDCでIAMロール polisjapan-gha-deploy-server をAssume（ECR pushとLambdaコード更新の権限のみ）
  2. コンテナイメージ2種をビルドしECRへpush（タグ = コミットSHA）
  3. aws lambda update-function-code で3関数を新イメージに切り替え
  4. /batch/healthcheck で疎通確認（コールドスタート込みでリトライ。失敗するとワークフローが赤くなる）
DockerfileECR リポジトリ使われる Lambda
Server/web/Dockerfile.lambdapolisjapan-apiapi / batch-update（同一イメージ・ハンドラ指定のみ変更）
Server/web/Dockerfile.lambda-chromiumpolisjapan-batch-createbatch-create（Selenium + headless Chromium）
⚠ Terraformとの役割分担 Lambdaの image_uri はTerraform側で ignore_changes 済み。イメージ更新はCIの仕事、インフラ定義はTerraformの仕事で、terraform apply がCIのデプロイを巻き戻すことはない。

デプロイ後の確認

  1. GitHub Actions のワークフローが緑であること（healthcheckまで含む）
  2. 変更内容に応じて: AI テーマ生成 E2E／管理画面の一覧・承認操作／CSV が5分以内に更新されること
  3. CloudWatch アラーム（batch-update / batch-create の Errors）が発火していないこと

手動デプロイ（Actions障害時のフォールバック）

# ECR ログイン
aws ecr get-login-password --region ap-northeast-3 --profile terraform \
  | docker login --username AWS --password-stdin 133078632695.dkr.ecr.ap-northeast-3.amazonaws.com

# ビルド & push（Server/web で実行。--provenance=false --sbom=false は必須:
# Lambdaはマルチプラットフォームマニフェスト非対応）
docker buildx build --platform linux/amd64 --provenance=false --sbom=false --push \
  -f Dockerfile.lambda -t 133078632695.dkr.ecr.ap-northeast-3.amazonaws.com/polisjapan-api:manual-YYYYMMDD .

# Lambda切り替え（batch-createも同様）
aws lambda update-function-code --profile terraform --function-name polisjapan-api \
  --image-uri 133078632695.dkr.ecr.ap-northeast-3.amazonaws.com/polisjapan-api:manual-YYYYMMDD

シークレットの取り扱い

環境シークレットの置き場所
本番（Lambda）SSM Parameter Store (SecureString) → Terraform が Lambda 環境変数へ注入
ローカル開発Server/web/.env（gitignore対象・雛形は .env.example）→ docker-compose が読み込み
GitHub Actionsシークレット保持なし。OIDCで一時クレデンシャルを取得（mainブランチのみAssume可）

Server/web/api/configs/ の設定ファイルはすべて環境変数から読む形式（2026-07-07〜）のため、コードにシークレットの実値は存在しない。

C. インフラ（Terraform）の変更

実行場所は Infra/terraform/environments/production/。tfstate は s3://polisjapan-tfstate（バージョニング有効）、実行には terraform プロファイル（terraform-deploy IAM ユーザー）を使う。

モジュール管理対象
modules/dataDynamoDB（drafts）/ アーカイブ S3 / ECR / SSM パラメータ
modules/apiLambda「api」+ API Gateway + ACM + カスタムドメイン
modules/batchLambda「batch-update / create」+ EventBridge Scheduler
modules/admin-siteadmin.pol-is.jp（S3 + CloudFront + IP 制限 Function）
modules/monitoringRoute53 ヘルスチェック + アラーム + SNS 通知

既存リソース（S3 / CloudFront / Route53 / ACM / contact・share Lambda / SES）も import 済みで、計 115 リソースがコード管理下。データを持つリソース（S3 / Route53 / DynamoDB）には prevent_destroy 設定済み。

⚠ シークレット値は Terraform に書かない SSM パラメータはガワだけ Terraform で作り、値は aws ssm put-parameter --type SecureString --overwrite で手動投入する（tfstate には Lambda 環境変数として復号値が残るため、tfstate バケットは非公開厳守）。

本番の自動実行（デプロイ後に動くもの）

ジョブ間隔内容
batch-update5 分pol.is からレポート CSV を取得し、S3 の themes.csv・レポート CSV を Cache-Control 付きで更新
batch-create15 分APPROVED の下書きを Selenium で pol.is に作成（1 回の起動で 1 件だけ処理）

起動は EventBridge Scheduler。失敗時は Lambda の Errors アラーム → SNS「Polis-Server-Alert」→ メール通知。
