## 🚨 1. 緊急対応事項（コスト以前の問題）

### 1.1 api.pol-is.jp の TLS証明書が期限切れ

- ヘルスチェックCanaryが **2026-05-07頃から約2ヶ月間、失敗し続けている**（`SEC_ERROR_EXPIRED_CERTIFICATE`）
- CloudWatchアラーム `Synthetics-Alarm-polis-japan-api-healthcheck-1` は **2ヶ月間 ALARM状態のまま**。再起動用Lambda（`PolisJAPAN-Reboot`）は状態遷移時にしか発火しないため、自動復旧は機能していない
- **影響**: WEBアプリからの「AIアシストによるテーマ作成」（`https://api.pol-is.jp/theme/*`）と管理画面がユーザーから利用不能。バッチ（cron→内部 `http://web:8000`）はEC2内部通信のため動作継続しており、投票数のCSV更新は生きている
- **対処**: EC2にSSHし `Server/docs/SSL証明書の更新手順.md` の certbot 更新手順を実施。cron による自動更新（`certbot renew`）が動いていない原因も確認が必要
- ※サーバーレス移行後は API Gateway + ACM になり、この証明書更新運用自体が消滅する

### 1.2 アラート通知が機能しているか確認

SNS `Polis-Server-Alert` → Gmail への通知購読はあるが、2ヶ月間の障害が放置されていたことから、通知が届いていない/埋もれている可能性が高い。移行後の監視設計で改善する。

## 5. サーバーレス移行設計案

### 5.1 目標

- EC2（$19.6）+ EBS（$6.1）+ IPv4（$3.6）+ 証明書運用を廃止
- 目標ランニングコスト: **月$5〜15（税抜）** — Route53 $0.5 + Lambda/API GW/DynamoDB無料枠内〜数ドル + S3/CloudFront微小 + OpenAI API費用（AWS外）
- 証明書はすべてACM自動更新化し、今回のような期限切れ事故を構造的に排除

### 5.2 コンポーネント対応表

| 現行（EC2上） | 移行先 | 備考 |
| --- | --- | --- |
| nginx + Let's Encrypt | **API Gateway（HTTP API）+ ACM** | api.pol-is.jp をカスタムドメイン化。証明書運用消滅 |
| FastAPI（theme/admin/batch） | **Lambda（コンテナ or zip）+ Mangum** | 既存FastAPIコードをほぼ無改修でLambda化できる。AI生成は現行タイムアウト300秒 < Lambda上限15分 |
| MySQL（t_draft） | **DynamoDB** 1テーブル | 実質使用は下書き1テーブルのみ。RDSを立てると本末転倒（最低$15〜/月）。`cruds/` 層の書き換えが必要 |
| MySQL（t_user系 3テーブル） | 廃止（未マウントの休眠機能） | 必要になったらDynamoDBに追加 |
| memcached | 廃止 | Lambda化で不要 |
| cron（5分毎 update / create_all） | **EventBridge Scheduler** → Lambda直接起動 | アクセスキーによる自己APIコールも不要になる |
| Seleniumバッチ（pol.is操作） | **Lambdaコンテナイメージ（headless Chromium）** | メモリ2GB程度で動作見込み。1テーマ作成が15分を超えるようならFargateスケジュールタスクにフォールバック |
| Twitter/Togetterスクレイピング | 同上（利用継続するなら） | 現在の利用状況を確認して取捨選択 |
| 管理画面（EC2 nginx `/dashboard/`） | S3 + CloudFront + **CloudFront Function or WAFでIP制限** | 静的ファイルなので他サイトと同じ構成に統一 |
| Synthetics Canary | **EventBridge Scheduler + 軽量Lambda（httpxでGET）+ アラーム** or 外部監視 | ブラウザ実行は不要（HTTPステータス確認で十分）。$21/月 → ほぼ$0 |
| CloudFront無効化 | 廃止（Cache-Control TTLで代替） | クイックウィン#1と同じ |

### 5.3 移行フェーズ

1. **Phase 0 — 応急処置**（即日）: TLS証明書更新、クイックウィン#1〜5 適用
2. **Phase 1 — IaC基盤**: AWS SAM（推奨・Lambda中心の構成に最適）でテンプレート作成。既存のcontact/share Lambdaも取り込んで手動管理から脱却
3. **Phase 2 — API移行**: FastAPI+Mangum をLambda化、`t_draft` をDynamoDBへ移行、API Gateway + ACMで `api.pol-is.jp` を切り替え（Route53のAレコードをエイリアスに変更するだけなので切り戻しも容易）
4. **Phase 3 — バッチ移行**: EventBridge Scheduler + Lambda（update系）、Seleniumバッチをコンテナイメージ化して検証
5. **Phase 4 — 撤収**: 動作確認後、EC2 2台・EBS・EIP・Canary・旧アラームを削除。管理画面をS3配信に移設

### 5.4 リスクと留意点

- **Seleniumバッチが最大の技術リスク**。pol.is のUI操作は実行時間・メモリのプロファイルを先に計測してからLambda/Fargateを選定する
- DynamoDB移行で `cruds/` `services/` のDB層書き換えが発生（影響は `t_draft` 関連のみ）
- Lambda化後のコールドスタート（LangChain読み込みが重い）: テーマ作成は対話的な機能のため、必要ならメモリ増量 or SnapStart的な対策を検討
- 移行中も現行EC2は並行稼働させ、Route53の切り替えで無停止移行する
